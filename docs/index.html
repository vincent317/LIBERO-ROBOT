Experiment loop

Every improvement came from a specific failure mode

I used rollout review to move from generic hyperparameter tuning toward targeted interventions: shorter action chunks, gripper-transition sampling, and a lower ACT KL penalty.

1

Get ACT running end to end

I converted LIBERO HDF5 demonstrations into a LeRobot image-backed dataset and wrote local training and eval entrypoints to work around environment and cache issues. The first stable recipe used two RGB cameras, robot state, and chunked action prediction.

2

Diagnose failed rollouts

The main failure was not the policy reaching the scene; it was grasp acquisition after contact. That pushed the next experiment toward gripper transition moments rather than broad data reweighting.

3

Sample the moments that matter

Oversampling both grasp and release transitions improved the single-task benchmark from 44% to 62%, showing that failure analysis was pointing at the right part of the trajectory.

4

Tune the objective, not just the data

Reducing ACT KL weight from 10.0 to 5.0 lifted success to 82%. Pushing it lower to 2.0 dropped success to 76%, so the best result came from an ablation, not from assuming monotonic improvement.

The lesson was not “oversample harder”

I tried task-level oversampling for weak tasks 04, 05, and 09. It looked plausible, but benchmark mean success collapsed from 69.4% to 22.0%. That result made the imbalance story less convincing and pushed the project toward cleaner architecture ablations.

Balanced batches

Even per-task batches isolated sampling effects, but the comparable benchmark result dropped to 61.4%. More balanced gradients did not automatically improve rollout success.

Shared task embedding

Adding the same task embedding to the ACT/VAE encoder, transformer encoder, and decoder trained stably and reached 65.8% at the comparable checkpoint.

VAE-only task ID

A narrower architecture ablation kept validation loss almost tied with the original baseline. It is the cleanest pending benchmark candidate because it changes one conditioning location.

Hard-case debugging

Negative results became useful evidence

The weakest multitask scenario was task 05: picking up a bowl on a ramekin. I trained a dedicated single-task policy with the best known single-task recipe, and it still reached 0 / 50. That pointed toward task-specific train/eval mismatch or data conversion issues rather than simple undertraining.

Engineering details

What I built around the model

Data conversion

Converted LIBERO demos into LeRobot datasets with agent-view RGB, wrist RGB, robot state, actions, and task IDs for multitask conditioning.

Local ACT training

Wrote local ACT-only trainers to avoid brittle import paths, redirect caches, support AMP, and log timing around DataLoader, host prep, and GPU steps.

Benchmark evaluation

Evaluated on LIBERO pruned initial states rather than demo starts, saved rollout videos, and tracked task-level success for the full spatial benchmark.
