Embodied AI MLInfra case study

Building and benchmarking LIBERO robot-learning training systems

This project turns LIBERO manipulation data into reproducible training and evaluation pipelines, then scales the work from single-task ACT policies to multitask ACT and OpenVLA fine-tuning on 4 x NVIDIA L4 GPUs. The emphasis is infrastructure: data conversion, policy training loops, distributed launch reliability, GPU memory profiling, throughput measurement, and benchmark evaluation.

What I built

Robot-learning stack, not only a policy result

The work spans two policy families. For LeRobot ACT, I built local conversion, training, and LIBERO benchmark evaluation paths for LIBERO-Spatial. For OpenVLA, I converted LIBERO HDF5 demonstrations into RLDS/TFDS datasets and benchmarked distributed LoRA strategies for the LIBERO-90 and LIBERO-100 training target.

ACT policy training

Converted LIBERO demos into image-backed LeRobot datasets with agent-view RGB, wrist RGB, robot state, actions, task text, and task IDs. Added ACT-only trainers to avoid brittle upstream import paths and local cache issues.

Benchmark evaluation

Evaluated policies in LIBERO off-screen simulation from pruned benchmark initial states, saved metrics and rollout videos, and added parallel evaluation for the 10-task spatial benchmark.

OpenVLA training infra

Added RLDS conversion and dataset registry support, fixed quantized DDP device placement, measured QLoRA/no-quant LoRA/FSDP behavior, and selected practical recipes for 4 x L4 training.

Track 1

LIBERO-Spatial single-task ACT

The first milestone was to make one ACT policy reliably solve a spatial manipulation task: pick up the black bowl and place it on the plate. I treated this as a closed training system problem: convert data, train, evaluate from benchmark init states, review failures, and run targeted ablations.

Track 2

LIBERO-Spatial 10-task ACT

The next step was one task-conditioned ACT policy across all ten LIBERO-Spatial tasks. This exposed the harder systems problem: mean success can look strong while specific tasks fail, so the evaluation stack needed per-task metrics, parallel workers, and controlled ablations.

Best 10-task mean 69.4%

500 benchmark rollouts

Scaling ACT

Task conditioning and evaluation throughput

  1. Converted all 10 LIBERO-Spatial tasks into one dataset: 500 episodes and 62,250 frames.
  2. Added learned discrete task IDs to the ACT encoder and decoder path.
  3. Kept the policy inputs compact: agent-view RGB, wrist RGB, robot state, and task_index.
  4. Parallelized benchmark evaluation by sharding tasks across worker processes.
  5. Used per-task success to identify weak variants instead of optimizing only the aggregate mean.

Track 3

LIBERO-100 / OpenVLA multi-GPU training infrastructure

For the larger embodied-AI model path, I moved from small ACT policies to OpenVLA-7B fine-tuning. The local LIBERO-100 target is represented as true LIBERO-90 plus LIBERO-10, not a synthetic spatial/object/goal merge. The work focused on data correctness, DDP reliability, memory headroom, and throughput tradeoffs on 4 x L4.

Recommended recipe DDP

4-bit LoRA, rank 32, effective batch 64

RLDS / TFDS conversion

Added an OpenVLA-compatible converter from official LIBERO HDF5 demonstrations to RLDS/TFDS. Verified smoke conversion and TFDS loading, then added dataset registry entries for libero_90_no_noops, libero_10_no_noops, and the libero_100_no_noops mixture.

Distributed launch fixes

Patched quantized model loading to use per-rank device maps and enabled DDP static graph behavior to avoid LoRA plus gradient-checkpointing repeated-ready errors. Verified 4-GPU `torchrun` smoke runs outside the sandbox.

Benchmark harness

Wrote short-run benchmark summaries with per-rank peak allocated/reserved memory, `nvidia-smi` peak memory, optimizer-step timing, effective batch size, and samples per second. This made strategy selection evidence-based rather than preference-based.

Why this is MLInfra work

Systems lessons I would bring to embodied-AI training

Data correctness first

The training result is only meaningful if conversion preserves image streams, action/state shape, task identity, and benchmark initialization semantics. I built conversion manifests and smoke checks before running longer training jobs.

Measure the actual bottleneck

Reducing LoRA rank from 32 to 16 halved trainable parameters, but did not materially reduce no-quant peak memory. The bottleneck was dominated by the 7B base model, DDP/activation/startup allocations, and memory reservation behavior.

Pick the recipe from evidence

On this hardware, DDP plus 4-bit LoRA is the robust path. FSDP reduced the conceptual model-sharding concern but was far slower in the measured setup, while no-quant LoRA left too little memory headroom for stable production runs.
